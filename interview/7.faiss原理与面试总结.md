## 1. FAISS 概述

### 1.1 什么是 FAISS

FAISS（Facebook AI Similarity Search）是 Facebook AI 团队开发的用于高效相似性搜索的库，特别适合处理大规模向量数据的索引和检索。

**核心特点：**
- 亿级向量索引能力
- 支持多种索引类型
- GPU 加速支持
- 纯 C++ 实现 + Python 接口

### 1.2 解决的问题

在推荐系统、搜索系统中，常见的向量化召回场景：
- embedding 向量检索
- 近似最近邻搜索（ANN）
- 百万/亿级向量库的实时检索

**暴力搜索的问题：** 当向量库达到百万级别时，暴力遍历 O(N) 的复杂度无法满足实时性要求。

## 2. FAISS 核心原理

### 2.1 索引构建流程

```
原始向量 → 量化(Quantization) → 构建索引 → 存储
```

**关键概念：**

| 概念 | 说明 |
|------|------|
| Vector | 待索引的向量，如 item embedding |
| Index | 索引结构，负责加速搜索 |
| Quantizer | 量化器，负责向量编码/解码 |
| IVF | 倒排索引，加速定位候选区间 |

### 2.2 量化压缩原理

FAISS 使用 **Product Quantization (PQ)** 技术将高维向量压缩为紧凑的编码。

**PQ 核心思想：**
1. 将向量切分为 m 个子向量（sub-vector）
2. 每个子向量独立进行 k-means 聚类（通常 k=256）
3. 用子向量所属的聚类中心 ID 作为编码
4. 最终用 m × 8 bits 表示原向量

```
原始向量维度: d
子向量个数: m
每个子向量编码: log2(k) bits (k=256 时为 8 bits)
最终编码长度: m × 8 bits
压缩率: d × 32 / (m × 8)
```

### 2.3 距离计算

FAISS 支持多种距离度量：

| 距离类型 | 说明 | 适用场景 |
|----------|------|----------|
| L2 距离 | 欧氏距离 | 通用场景 |
| IP (Inner Product) | 内积/余弦相似度 | 向量归一化后的余弦相似度 |

**注意：** 使用内积作为距离时，通常需要先对向量做归一化处理。

## 3. 索引类型详解

### 3.1 索引分类

```
FAISS Index
├── Flat
│   ├── IndexFlatL2 (暴力搜索，无压缩)
│   └── IndexFlatIP (内积暴力搜索)
├── IVF (倒排索引)
│   ├── IndexIVFFlat
│   ├── IndexIVFPQ
│   └── IndexIVFScalarQuantizer
├── LSH ( locality-sensitive hashing)
│   └── IndexLSH
├── HNSW (分层可导航小世界图)
│   └── IndexHNSW
└── Composite (组合索引)
    ├── IndexIDMap
    └── IndexPreTransform
```

### 3.2 常用索引详解

#### Flat 索引

**IndexFlatL2 / IndexFlatIP**

- 最简单的索引，无任何压缩和加速
- 精确检索（exact search）
- 适用场景：小规模数据（<1万），或作为其他索引的 baseline

```python
import faiss
dim = 128
index = faiss.IndexFlatL2(dim)  # 或 IndexFlatIP(dim)
index.add(data)  # 添加向量
D, I = index.search(query, k)  # 返回距离和索引
```

#### IVF 索引 (Inverted File Index)

**原理：** 对向量进行聚类，搜索时先定位最近的几个聚类中心，再在候选聚类中搜索。

```python
nlist = 100  # 聚类中心数
quantizer = faiss.IndexFlatL2(dim)
index = faiss.IndexIVFFlat(quantizer, dim, nlist)
index.train(data)  # 训练聚类中心
index.add(data)
index.nprobe = 10  # 查询时探索的聚类数（影响召回率和速度）
```

**关键参数：**
- `nlist`: 聚类中心数量（通常设为 4×√N）
- `nprobe`: 查询时探索的聚类数（越大越准确，但越慢）

#### PQ 索引

**Product Quantization + IVF 组合：**

```python
m = 16  # 子向量个数
nlist = 100  # 聚类中心数
index = faiss.IndexIVFPQ(quantizer, dim, nlist, m, 8)  # 8 bits per sub-vector
```

**适用场景：** 内存受限的大规模向量检索

#### HNSW 索引

**原理：** 构建分层可导航小世界图，搜索时从顶层快速定位到目标区域。

```python
M = 32  # 每层连接数
efConstruction = 200  # 构建时的搜索范围
index = faiss.IndexHNSWFlat(dim, M)
index.hnsw.efConstruction = efConstruction
```

**参数说明：**
- `M`: 节点连接数（越大越准确，内存越大）
- `efSearch`: 搜索时的遍历队列大小（越大越准确，越慢）
- `efConstruction`: 构建时的参数

**特点：**
- 无需训练
- 内存占用相对较高
- 构建慢，查询快

## 4. 工程实践

### 4.1 索引选择指南

| 数据规模 | 推荐索引 | 备注 |
|----------|----------|------|
| < 1万 | Flat | 暴力搜索，精确结果 |
| 1万 ~ 100万 | IVF 系列 | 平衡精度和速度 |
| 100万 ~ 10亿 | IVF + PQ | 内存优先 |
| 任意规模，高召回 | HNSW | 查询速度优先 |

### 4.2 参数调优经验

**IVF 索引调优：**
```python
# 1. 选择合适的 nlist
nlist = 4 * int(math.sqrt(n_vectors))  # 经验公式

# 2. 查询时调整 nprobe
# 召回率优先: nprobe = nlist 或更大
# 速度优先: nprobe = 1~10
index.nprobe = 10
```

**HNSW 索引调优：**
```python
# 精度优先
index.hnsw.efSearch = 256  # 甚至更大

# 速度优先
index.hnsw.efSearch = 16
```

### 4.3 GPU 加速

```python
res = faiss.StandardGpuResources()
index = faiss.IndexFlatL2(dim)
gpu_index = faiss.index_cpu_to_gpu(res, 0, index)
```

### 4.4 工程注意点

1. **向量归一化：** 使用内积距离前务必归一化
2. **维度对齐：** batch add 时注意 dim 的一致性
3. **内存预估：** PQ 索引内存 ≈ 原始数据 × (1 + 1/m)
4. **索引持久化：** `faiss.write_index()` / `faiss.read_index()`

## 5. 面试高频问题

### 5.1 基础概念类

**Q1: 简述 FAISS 的工作原理**

FAISS 是一种近似最近邻（ANN）搜索库，核心通过量化（Quantization）和索引（Index）技术加速向量检索。原始向量被压缩为紧凑编码，搜索时无需遍历全量数据，只需在候选区间内搜索。

**Q2: FAISS 和暴力搜索的区别是什么？**

| | 暴力搜索 | FAISS |
|--|---------|-------|
| 复杂度 | O(N) | O(log N) ~ O(N/k) |
| 召回率 | 100% | 可调，通常 95%+ |
| 内存 | 原始向量 | 可压缩到 1/10 |
| 速度 | 慢 | 快 |

**Q3: 什么是 Product Quantization？**

PQ 将高维向量切分为多个子向量，每个子向量独立量化（通常量化到 256 个类），最终用子向量的类 ID 组成紧凑编码。例如 128 维向量切分为 16 个 8 维子向量，每个用 8 bits 编码，总共 128 bits。

### 5.2 索引原理类

**Q4: IVF 索引的原理是什么？**

IVF（倒排文件索引）首先对全量向量进行聚类（如 k-means），每个聚类中心管理属于它的向量。搜索时：
1. 计算 query 与所有聚类中心的距离
2. 选择最近的 nprobe 个聚类
3. 在选中的聚类中遍历搜索

**Q5: HNSW 和 IVF 的区别？**

| | HNSW | IVF |
|--|------|-----|
| 结构 | 图结构 | 聚类结构 |
| 构建 | 慢，需建图 | 快，只需聚类 |
| 查询 | 快，稳定 | 依赖 nprobe |
| 内存 | 较大 | 较小（可配合PQ） |
| 召回率 | 高 | 可调 |

**Q6: nprobe 参数对结果有什么影响？**

nprobe 控制搜索时探索的聚类数量。nprobe=1 只会搜索最近的聚类，速度快但召回率低；nprobe=nlist 会搜索所有聚类，等同于暴力搜索。通常在召回率和速度间做权衡。

### 5.3 实践应用类

**Q7: 如何选择合适的索引类型？**

- 小规模数据（<1万）：直接用 Flat
- 中等规模，需要精度：IVFFlat
- 大规模，内存受限：IVFPQ
- 大规模，高召回要求：HNSW
- 超大规模，精度/内存平衡：IVFPQ + 增大 nprobe

**Q8: FAISS 在推荐系统中的应用场景？**

1. **向量召回：** item embedding 存储与检索
2. **多兴趣召回：** MIND 等多兴趣网络产出的多个向量，每个向量独立检索
3. **图召回：** DeepWalk/Node2Vec 产出的图 embedding 检索
4. **实时更新：** 支持增量 add，但删除需重建索引

**Q9: FAISS 的局限性有哪些？**

1. **删除操作：** 不支持高效删除，需重建索引
2. **动态更新：** 增量更新后需要判断是否需要重新训练
3. **Filter 限制：** 原生不支持带条件的过滤，需要体外实现
4. **分布式：** 单机版，多机需要额外封装

**Q10: 如何提升 FAISS 的召回率？**

1. 增大 nprobe（IVF）或 efSearch（HNSW）
2. 增大 PQ 的子向量数 m 和聚类数 k
3. 增大 HNSW 的 M 和 efConstruction
4. 使用更高效的索引类型（如 HNSW 替换 IVF）
5. 适当降级压缩率

### 5.4 开放问题类

**Q11: Faiss 如何判断当前索引是否需要重新训练？**

当新增向量分布与训练数据差异较大时需要重新训练。可以通过监控召回率或在新旧数据上分别测试来判断。

**Q12: 有哪些开源的向量数据库？各自特点？**

| 向量库 | 特点 |
|--------|------|
| Milvus | 功能完善，支持分布式 |
| Pinecone | 云原生，托管服务 |
| Weaviate | 原生支持混合搜索 |
| Qdrant | 性能好，支持 Filter |
| Chroma | 轻量级，面向 LLM |
